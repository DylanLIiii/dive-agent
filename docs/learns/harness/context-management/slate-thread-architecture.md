# Slate: 超越 ReAct 和 RLM 的代理架构

> **Related topics**: [[session-history-management]], [[codex-llm-abstraction]], [[llm-framework-comparison]]

## 概述

Slate 是 Random Labs 提出的一种新型代理架构，旨在解决现代 LLM 代理面临的三个核心问题：长时任务执行、策略与战术的平衡、以及工作内存管理。

**项目地址**: https://randomlabs.ai/blog/slate

---

## 1. 现代代理面临的三大难题

### 1.1 长时任务 (Long-Horizon Tasks)

长时任务是**路径依赖**的——完成任务所需的最少步骤超出了最小 harness 的能力。

最小 harness 定义为：仅包含有限工具调用循环的模型，无额外规划或内存基础设施。

解决此类任务需要三个要素：
- 足够的工作内存，使模型能在正确的时间关注正确的上下文
- 战略与战术执行的平衡，使模型既能良好规划又能正确执行
- 在不丢失总体目标的前提下，整合任务过程中发现的新信息

### 1.2 工作内存 (Working Memory)

模型无法在整个上下文窗口中均匀关注信息。随着上下文长度增长，模型关注信息的能力会下降。

**Dumb Zone**: Dex Horthy 提出，指上下文窗口中检索质量下降的区域。工作内存实际上是到该点为止的上下文窗口。

```
+------------------------------------------+
|          Context Window                   |
+--------------------+---------------------+
|   Working Memory   |     Dumb Zone       |
| (usable attention) | (degraded quality)   |
+--------------------+---------------------+
```

研究显示，所有前沿模型在输入长度增加时都会出现非均匀的性能下降。

### 1.3 策略与战术 (Strategy vs. Tactics)

| 概念 | 定义 |
|------|------|
| **策略 (Strategy)** | 基于知识的开放性规划，引导系统走向目标 |
| **战术 (Tactics)** | 学到的、局部的动作序列，帮助实际迈向目标 |

软件工程任务涵盖这两个维度：
- 战术：运行 bash 命令、写测试用例
- 战术：设计模式使其向后兼容

## 2. 现有方案的局限

| 方案 | 解决的问题 | 失败模式 |
|------|----------|----------|
| **Compaction (压缩)** | 工作内存 | 非确定性丢失、难以可靠压缩 |
| **Subagents (子代理)** | 上下文隔离 | 跨边界信息传递困难 |
| **Markdown Planning** | 战略连贯性 | 计划不充分、执行不完整、忘记更新计划 |
| **Task Trees** | 早期停止 | 刚性、难以适应新信息、表达性低 |
| **RLM** | 灵活分解 | 过度分解、缺乏中间反馈 |
| **Devin/Manus/Altera** | 长期状态 | 压缩边界丢失关键状态 |
| **Claude Code/Codex** | 简单委托 | 消息传递同步问题 |

### 2.1 关键洞察：知识 overhang

模型的潜在知识覆盖了任务空间中的广泛轨迹，但直接战术采样只能访问狭窄区域。规划、思维链和脚手架扩展了采样区域。

```
知识可访问性
     ^
     |    /\
     |   /  \ 知识 overhang
     |  /    \_______________
     |/
     +-------------------------> 任务复杂度

     ■ 战术可访问    ■ 潜在知识
```

## 3. Slate 架构：线程与片段

### 3.1 核心概念

**Thread (线程)**:
- 一个高度表达性的接口，用于访问模型的知识 overhang
- 中央编排代理使用 DSL 将动作委托给工作线程
- 每个线程执行一个动作，完成后暂停并将控制权交回主线程
- 可看作是"战术"：运行命令序列、从文件 Y 中提取 X 等

**Episode (片段)**:
- 线程完成动作时所采取步骤的压缩表示
- 真正的情景记忆形式：只保留重要结果，不保留完整的战术轨迹
- 片段可以直接作为其他线程的输入，实现线程的组合性

### 3.2 线程编织 (Thread Weaving)

线程编织的结果是一个**隐式且自适应**分解任务的系统——无需静态计划。

机制：
1. 编排器通过引用使用线程
2. 给予复杂上下文路由的语义
3. 线程范围有界，系统自然与当前计划同步
4. 线程由 LLM 驱动而非静态脚本，可响应意外状态

```
编排器
   │
   ├────▶ 线程 T1 ──▶ 片段 ──┐
   │                        │
   ├────▶ 线程 T2 ──▶ 片段 ──┼──▶ 输入到后续工作
   │                        │
   └────▶ 线程 T3 ──▶ 片段 ──┘
```

### 3.3 与子代理的对比

| 特性 | 子代理 | 线程 |
|------|--------|------|
| 上下文隔离 | 各自独立的隔离气泡 | 显式共享/可组合 |
| 通信方式 | 消息传递 | 显式上下文传递 |
| 持久性 | 持久，有时后台运行 | 仅累积特定工作流的上下文 |
| 返回值 | 响应消息 | 压缩的片段历史 |

## 5. OS 视角：LLM 作为操作系统

Slate 的线程架构直接映射到 Karpathy 提出的 LLM OS 框架：

| OS 概念 | LLM 对应 |
|---------|---------|
| 内核 (Kernel) | 编排层 |
| 进程 (Processes) | 线程 |
| 进程返回值 | 片段 (压缩的工作总结) |
| RAM | 上下文窗口 |
| 外设 | 文件系统、终端、浏览器 |

**关键洞察**：长时代理任务的真正瓶颈是**上下文管理**，而非模型智能。模型已经足够聪明能解决更多任务，差距是系统问题，不是能力问题。

## 6. 有趣的观察

1. **大规模并行执行**：真实软件任务自然分解为并行线程工作流，编排器可以同时调度多个线程

2. **跨模型组合**：在同一任务中使用 Sonnet 和 Codex 效果良好，片段边界充当模型间的清晰交接

| 组件 | 技术 |

## 8. 关键洞察

1. **问题相互作用**: 长时任务、策略/战术、内存管理这三个问题相互影响，必须同时解决

2. **表达性很重要**: Harness 的表达性决定了模型能到达的行为空间

3. **知识 overhang**: 模型知道的比它能直接访问的多，规划是解锁潜在知识的关键

4. **有界同步**: 频繁、有界的编排器与工作线程同步提供了速度、延迟和智能的平衡

5. **片段即记忆**: 片段是真正情景记忆的可处理形式，压缩边界自然
